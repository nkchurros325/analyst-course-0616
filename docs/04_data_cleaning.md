# 04. データクレンジング / 前処理

> このモジュールのゴール：**わざと汚した顧客データ `customers_raw.csv` を、分析に使える「きれいな顧客マスタ」に整える。**
> 「現実のデータは汚い」という前提に立ち、汚れを**発見し、理由を持って判断し、記録に残す**一連の流れを身につけます。

---

## 4-1. なぜ前処理にこだわるのか

> **現実のデータは、ほぼ例外なく汚い。**

入力ミス、表記ゆれ、システム間の連携バグ、人手の集計 ── 原因はさまざまですが、
そのまま集計すると平気で間違った結論が出ます。アナリストの実務では
**「前処理（クレンジング）に分析時間の半分以上」**かかることもざらです。地味ですが、ここが分析の土台です。

そして前処理で最も大事なのは作業の速さではなく、**判断に理由があること**です。

> 💡 合言葉は **「クレンジングの判断には、必ず理由を」**。
> 「異常値だから消した」ではなく「年齢999は入力上ありえない（人間の年齢の上限を超える）から欠損として扱い、中央値で補完した」と説明できる状態にします。
> あとで「なぜこの行が消えたのか」を自分でも他人でも追えることが、信頼される分析の条件です。

---

## 4-2. クレンジングの全体像

汚れの種類はだいたい次の5つに整理できます。本モジュールはこの順で潰していきます。

| 汚れの種類 | 例（今回のデータ） | 主な対処 |
|------------|--------------------|----------|
| 欠損値 | `age` や `gender` が空 | 削除 or 補完（fillna / dropna） |
| 重複 | 同じ顧客が複数行 | `drop_duplicates` |
| 異常値（外れ値） | `age` が -1, 0, 200, 999 | 範囲外を欠損化 → 補完 |
| 表記ゆれ | `gender` に 女性 / F / M が混在 | マッピングで統一 |
| 型・フォーマット | 日付が `-` と `/` 混在、空白混入 | `to_datetime`、`str.strip` |

進め方の原則は次の3ステップです。

```
① 発見する        →  ② 判断して処理する      →  ③ 検証する
   どこがどう汚い?      何を残し何を直す?(理由)     本当にきれいになった?
```

**②の前に必ず①をやる**こと。汚れを把握せずにいきなり `dropna()` すると、何件・何を捨てたか分からなくなります。

---

## 4-3. ① 汚れを「発見」する

最初にやるのは加工ではなく**観察**です。次の5つの道具で全体像をつかみます。

| メソッド | 何が分かるか |
|----------|--------------|
| `df.info()` | 行数・列・dtype・非欠損数。型がおかしい列に気づける |
| `df.isna().sum()` | 列ごとの欠損数 |
| `df.duplicated().sum()` | 完全重複している行数 |
| `df["col"].value_counts(dropna=False)` | カテゴリ列の表記ゆれ・欠損 |
| `df.describe()` | 数値列の min/max/平均。異常値の当たりがつく |

```python
df.isna().sum()             # 欠損の所在
df.duplicated().sum()       # 重複の数
df["gender"].value_counts(dropna=False)   # 表記ゆれを目視
df["age"].describe()        # min が -1、max が 999 → 異常値の気配
```

> 💡 `describe()` で `age` の `max` が 999、`min` が -1 のように**常識外の値**が出たら、ほぼ異常値です。
> いきなり平均を信じず、まず min/max を見る癖をつけましょう（外れ値は平均を簡単に歪めます）。

---

## 4-4. ② 欠損値の扱い ── 削除 vs 補完

欠損には大きく2つの対処があります。**どちらが正解かは「列の役割」と「欠損の量・理由」で決める**ものです。

| 方針 | メソッド | 向いている場面 | 注意点 |
|------|----------|----------------|--------|
| 削除 | `dropna()` | 欠損がごく少数 / その行は分析に使えない / キー列が欠損 | データ件数が減る。**捨てる前に必ず件数を確認** |
| 補完 | `fillna()` | 行は活かしたい / もっともらしい代表値で埋められる | 埋めた値は「推定」。事実ではないと意識する |

**数値の補完では、平均より中央値（median）が無難**なことが多いです。平均は外れ値に引きずられますが、
中央値は影響を受けにくいためです（今回のように 999 のような異常値が混ざるデータでは特に）。

```python
# 方針A：欠損行を捨てる（件数の変化を必ず確認）
before = len(df)
clean = df.dropna(subset=["age"])
print(before, "→", len(clean))   # 何件落ちたかを記録

# 方針B：中央値で補完する（行は残す）
med = df["age"].median()
df["age"] = df["age"].fillna(med)
```

> 💡 **判断軸**：その列がないと分析が成立しない（例：注文に customer_id が無い）なら削除寄り。
> 行は活かしたいし、欠損が「たまたま未記入」程度ならば補完寄り。
> どちらにせよ **「何件を、なぜ、どう処理したか」を記録**しておくこと。これが 4-1 の「理由を残す」です。

---

## 4-5. ③ 重複の除去

同一顧客が何度も登録される、抽出スクリプトのバグで行が二重になる ── 重複はよくある汚れです。
完全に同じ行は `drop_duplicates()` で1件に畳めます。

```python
print("重複行:", df.duplicated().sum())
df = df.drop_duplicates()          # 完全一致の重複を1件に
# 「customer_id だけ重複」を消すなら subset を指定
df = df.drop_duplicates(subset=["customer_id"], keep="first")
```

> 💡 「完全重複」と「キーの重複」は別物です。`customer_id` が同じでも他の列が違うなら、
> それは更新履歴かもしれません。**どのキーで重複とみなすか**を決めてから消しましょう。

---

## 4-6. ④ 異常値の検出と処理

異常値（外れ値）は「物理的にありえない値」と「統計的に極端な値」に分けて考えます。

- **ありえない値**：年齢 -1, 0, 999 など。ドメイン知識で明らかにおかしい → 欠損化して補完するのが安全。
- **統計的に極端な値**：他より突出して大きい/小さい値。判断には基準が要ります。代表的なのが **IQR法**です。

今回は年齢を **0〜120 の範囲外を欠損(NaN)に置き換え、そのうえで中央値で補完**します。
「消す」のではなく「いったん欠損扱いにして補完に回す」と、4-4 の欠損処理に流れを一本化できます。

```python
# 範囲外（0〜120 の外）を欠損化 → 中央値補完
df.loc[(df["age"] < 0) | (df["age"] > 120), "age"] = np.nan
df["age"] = df["age"].fillna(df["age"].median())
```

> 📎 **IQR法（軽い紹介）**：第1四分位 Q1 と第3四分位 Q3 から `IQR = Q3 - Q1` を求め、
> `Q1 - 1.5×IQR` 未満 / `Q3 + 1.5×IQR` 超を外れ値とみなす方法です（箱ひげ図のヒゲの基準）。
> ```python
> q1, q3 = df["age"].quantile([0.25, 0.75])
> iqr = q3 - q1
> low, high = q1 - 1.5 * iqr, q3 + 1.5 * iqr
> ```
> ただし IQR法は機械的なので、**消す前に「本当に異常か」をドメインで確認**します。
> 「高齢の優良顧客」を外れ値として一律に消すと、大事な層を取りこぼします。

## 4-7. ⑤ 文字列の正規化と表記ゆれの統一

人が入力した文字列には、見えない汚れが潜みます。

- **前後の空白**：`" 東京都"` と `"東京都"` は別物として集計されてしまう → `str.strip()`。
- **全角空白（`　`, U+3000）**：半角の `strip()` では落ちないことがある → 明示的に置換 or 指定。
- **表記ゆれ**：`女性 / 男性 / 回答なし` のほか `F / M / 空文字` が混在 → **マッピング辞書で統一**。

```python
# 前後の空白（半角・全角）を除去
df["prefecture"] = df["prefecture"].str.strip().str.replace("　", "", regex=False)
df["name"] = df["name"].str.strip().str.replace("　", "", regex=False)

# gender を3カテゴリに統一
gender_map = {"女性": "女性", "F": "女性", "男性": "男性", "M": "男性",
              "回答なし": "回答なし", "": "回答なし"}
df["gender"] = df["gender"].str.strip().map(gender_map).fillna("回答なし")
```

> 💡 マッピングは **「想定した値だけ」を明示**するのがコツ。`map()` は辞書に無い値を `NaN` にするので、
> 取りこぼし（想定外の表記）を `.isna()` で発見できます。最後に `fillna("回答なし")` で受け皿を作っておくと安全です。

> 📌 **正規化のあと、もう一度 dedup する**：`" 東京都"` と `"東京都"` のように
> 空白だけ違っていた行は、`strip` で揃った瞬間に**完全一致の重複**になることがあります。
> そのため実務では「正規化 → 重複除去」をこの順で（必要なら重複除去を2度）回します。
> 最初に1度 `drop_duplicates()` しただけで安心しないこと。

## 4-8. ⑥ 型変換 ── 日付フォーマットの混在

`registration_date` は `2023-12-24`（ハイフン）と `2024/06/04`（スラッシュ）が混在しています。
文字列のままでは並べ替えも期間フィルタもできないので、**datetime 型に統一**します。

混在フォーマットは pandas 2系の `format="mixed"` で素直にパースできます。
万一パースに失敗する値があっても落ちないよう、`errors="coerce"`（失敗を NaT にする）を併用すると堅牢です。

```python
df["registration_date"] = pd.to_datetime(
    df["registration_date"], format="mixed", errors="coerce"
)
# 変換できなかった行（NaT）が無いか確認
print("変換失敗:", df["registration_date"].isna().sum())
```

> 💡 `errors="coerce"` は「失敗したら NaT にして処理を続ける」オプション。
> 黙って欠損化されるので、**変換後に NaT の数を必ず確認**します。0件なら全行ちゃんとパースできた証拠です。

---

## 4-9. ⑦ 仕上げ ── クレンジング後の検証

「直したつもり」で終わらせないこと。**処理前と同じ道具でもう一度確認**し、汚れが消えたかを数字で示します。

```python
print("欠損:\n", df.isna().sum())          # → すべて 0 が理想
print("重複:", df.duplicated().sum())        # → 0
print(df["gender"].value_counts())          # → 3カテゴリだけ
print(df["age"].describe())                 # → min/max が常識範囲に
print("行数:", before_rows, "→", len(df))   # → 何件減ったかを報告
```

> 💡 **「捨てる前に件数を確認、捨てた後も件数を報告」**。
> 行数が `2,000 → 1,950` のように変わったら、その50件が何だったか（重複か欠損か）を言えるようにします。
> 黙って減った行数は、あとで「あれ、数が合わない」の原因になります。

## 4-10. 実務の心得まとめ

- **現実のデータは汚い**。前処理は分析の前座ではなく、分析の質を決める本体です。
- **発見 → 判断 → 検証**の順を守る。観察せずにいきなり加工しない。
- **クレンジングの判断には必ず理由を。** 「なぜ消したか/埋めたか」を説明できる状態にする。
- **捨てる前に件数を確認**し、**処理の記録を残す**（どの列を、何件、どうしたか）。
  - 実務ではクレンジング処理自体をコード（このノートブックのように）で残し、再現できるようにします。手作業でExcelをいじると後で再現できません。
- 補完した値は「推定」であって事実ではない。**埋めたことを忘れない**。
