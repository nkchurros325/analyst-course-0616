> 前提：モジュール01〜08。特に 01（思考法）の5ステップをこの章で実際になぞります。

## 9-1. ケースの背景

あなたは架空EC「**Sora Mart**」のデータアナリストです。ある日、経営からこう相談されました。

> 「最近、**売上の伸びが鈍化している気がする**。
> ① 現状を把握し、② 伸び悩みの要因仮説を立て、③ 次に打つべき施策を提案してほしい。
> 特に『どの**顧客層・経路・カテゴリ**に注力すべきか』『先日の **A/Bテスト（ボタン色）** の結果は本番投入してよいか』を知りたい。」

これは典型的な「ふわっとした相談」です。アナリストの最初の仕事は、これを**データで答えられる問い**に翻訳することです。

> 🧭 この章は「正解が1つに決まらない」リアルな演習です。
> 大事なのは**手順（型）をなぞること**と、**示唆と提案まで言い切ること**、そして**不確実性を正直に添えること**。

## 9-2. 分析プロジェクトの進め方（モジュール01の再掲）

実務の分析は、次の5ステップを回ります。**手を動かす前に①②を固める**のがコツでした。

```
① 問いを定義   →  ② 設計      →  ③ 取得・加工   →  ④ 分析・解釈   →  ⑤ 伝達
   何を知りたい?     どう測る?       SQL/pandas       統計/可視化       示唆と提案
```

このケースに当てはめると：

| ステップ | このケースでの中身 |
|---------|------------------|
| ① 問い | 「売上の伸びは本当に鈍化したか？ 鈍化したなら、客数・客単価・頻度のどれが効いているか？ 注力すべきセグメントは？ A/Bテストは本番投入してよいか？」 |
| ② 設計 | KPIを定義（下記）。比較軸を決める（前年比・MoM・セグメント間・A vs B）。 |
| ③ 取得・加工 | 4テーブルを merge し、`completed` のみで分析用データセットを作る。 |
| ④ 分析・解釈 | 推移→要因分解→セグメント→リピート→A/Bテスト→回帰。 |
| ⑤ 伝達 | 経営向けサマリー（結論ファースト）。 |

### KPI（指標）の定義 — ここを先に決めるのが肝心

| 指標 | 定義（このケースでの計算） |
|------|--------------------------|
| 売上（GMV） | Σ(`quantity × unit_price`)、**`status='completed'` のみ** |
| 注文数 | `completed` の注文（`order_id`）数 |
| 購入顧客数 | `completed` の注文を持つ顧客の人数 |
| 客単価（AOV） | 売上 ÷ 注文数 |
| 購入頻度 | 注文数 ÷ 購入顧客数 |
| リピート率 | 2回以上購入した顧客 ÷ 購入顧客数 |
| CVR（A/Bテスト） | converted=1 の数 ÷ ユーザー数 |

> 💡 **「売上が落ちた」を語るには分解が要る**：`売上 = 購入顧客数 × 購入頻度 × 客単価`。
> どの項が効いているかで、打つ手（獲得／リピート促進／単価アップ）が変わります。

## 9-3. 分析の流れ（ノートブックの構成）

ノートブック `09_capstone.ipynb` は次の順で進みます。各セクションは **🖊 課題（あなたが分析する）** 形式で、スケルトンを置いています。最後に **✅ 解答例** と **経営向けサマリー** があります。

1. **課題設定** … ビジネス課題を問いに翻訳し、KPIを定義する。
2. **データ準備** … 4テーブルを merge し、`completed` だけの「分析用ロングテーブル」を作る。
3. **現状把握** … 月次・年次の売上推移、前年比、MoM、季節性を可視化。
4. **要因分解** … `売上 = 客数 × 頻度 × 客単価`。さらに経路別・カテゴリ別・地域別・年代別で比較。
5. **リピート/離脱** … 新規 vs リピート顧客、購入回数の分布。
6. **A/Bテストの判断** … `ab_test.csv` をカイ二乗検定し、本番投入の是非を結論。
7. **回帰での補足** … 顧客の購入額を説明する要因を、scipy/numpy でざっくり。
8. **統合** … 結論→根拠→提案を経営向けサマリーにまとめる。

> 🧭 「鈍化している気がする」は仮説にすぎません。**まず推移を描いて事実確認**してから要因に進みます。
> 描いてみると「全体は伸びているが、直近の伸び率（MoM）が頭打ち」のような“ねじれ”が見えることがあります。
> そのときは経営の言葉（伸び悩み）とデータの事実（伸び率の鈍化）を**正直にすり合わせて**伝えるのがプロです。

## 9-4. 成果物としての「分析レポートの書き方」

総合演習のゴールは「コードが動いた」ではなく、**意思決定者が動けるレポート**です。次の3原則を守ります。

### ① 結論ファースト（PREP＋提案）

やった順ではなく、相手が知りたい順に。

```
【結論】 売上は前年比では大きく伸びたが、直近半年の月次の伸び率は頭打ち。
        伸びの主因は新規獲得で、客単価はほぼ横ばい。
【根拠】 月次推移グラフ／要因分解（客数 vs 頻度 vs 単価）／経路別・カテゴリ別の構成。
【示唆】 新規一本足では成長が鈍る。リピート顧客が売上の大半を支えている。
【提案】 ①伸びている経路に集中、②上位カテゴリの在庫・訴求を厚く、
        ③リピート促進施策、④A/Bテストのボタン色Bは有意に勝ったので本番投入。
```

### ② 1メッセージ1グラフ

1枚のグラフで言いたいことは1つに絞る。タイトルに**結論**を書く（「月次売上推移」ではなく「直近半年は月次の伸びが頭打ち」）。

### ③ 提案と不確実性をセットで

- 必ず**次のアクション**まで書く。「〜が分かった」で止めない。
- 同時に**限界**も書く：「ただしこのデータは購入者のみで、離脱者・非購入者は含まない」「観測データなので因果は断定できない」。

> 💡 **チェックリスト（提出前に自問）**
> - [ ] 問いに**一文**で答えられるか？
> - [ ] 売上は `completed` のみで計算したか？（cancelled/returned を混ぜていないか）
> - [ ] 数字に**比較対象**があるか？（前年・前月・他セグメント）
> - [ ] グラフのタイトルが「結論」になっているか？
> - [ ] A/Bテストは **p値・リフト・ビジネス判断** をセットで述べたか？
> - [ ] **提案**まで言い切ったか？ **不確実性**を添えたか？
> - [ ] 「この数字、他の説明はつかない？」を一度自問したか？

## 9-6. 次に学ぶとよいこと

この講座はアナリストの「型」を一通りなぞりました。ここから先は、興味と業務に応じて深掘りしましょう。

- **機械学習（予測）** … 「来月の売上は？」「この顧客は離脱しそうか？」を予測する。
  まずは `scikit-learn` の回帰・分類・クラスタリングから。本講座で禁止していた sklearn/statsmodels はこの先で活躍します。
- **ダッシュボード／BIツール** … 分析を「一度きり」で終わらせず、関係者がいつでも見られる形に。
  Tableau / Looker Studio / Power BI、あるいは Python の `streamlit` など。
- **因果推論** … 「相関」ではなく「効果」を測る。A/Bテスト（RCT）の設計、差分の差分法（DID）、傾向スコアなど。
  「施策をやったから売上が伸びたのか？」に答える力です。
- **データ基盤・SQLの深掘り** … ウィンドウ関数、CTE、dbt、データウェアハウス（BigQuery/Snowflake）。
  大規模データを扱う実務では SQL 力がそのまま生産性になります。
- **指標設計・実験文化** … 何をKPIに置くか、どうA/Bテストを回すか。技術以上に組織で効いてきます。

> 💡 最後にもう一度。アナリストの価値は **「意思決定を助けたか」** で決まります。
> 手法を増やすほど「で、何が言えるの？」に立ち返る癖を大切にしてください。
